# Лабораторная работа  
## Использование паттерна проектирования Memento

---

## Описание предметной области

В работе реализуется упрощённая игра в стиле Flappy Bird на библиотеке pygame.  
Пользователь управляет птицей, уклоняясь от труб и набирая очки. Система должна поддерживать:

- сохранение текущего состояния игры в один из слотов;
- восстановление ранее сохранённого состояния;
- удаление сохранений;
- отображение информации о каждом сохранении (очки и время);
- продолжение игры с последнего сохранения.

Состояние игры включает несколько взаимосвязанных данных: положение птицы, её скорость, текущий набор труб и счёт — что делает прямое копирование и восстановление состояния нетривиальной задачей.

---

## Решение

Для реализации системы сохранений был применён поведенческий паттерн **Хранитель (Memento)**.  
Созданы следующие участники паттерна:

- `Memento` — хранит снимок состояния игры: положение птицы (`bird_y`), скорость (`velocity`), список труб (`pipes`), счёт (`score`) и время создания снимка (`time`);
- `Game (Originator)` — владелец состояния; умеет создавать снимок через `create_memento()` и восстанавливаться из него через `set_memento()`;
- `SaveManager (Caretaker)` — хранит до 5 слотов с объектами `Memento`, не зная об их внутренней структуре.

Клиентская часть (игровой цикл) взаимодействует с `SaveManager` и `Game`, не занимаясь напрямую копированием или восстановлением данных — вся эта логика инкапсулирована внутри паттерна.

## Сравнение реализации с паттерном и без паттерна

### Реализация с использованием Memento

- класс `Game` самостоятельно формирует снимок своего состояния и восстанавливается из него;
- `SaveManager` хранит непрозрачные объекты `Memento`, не зная об их содержимом;
- внутренние поля `Game` не доступны снаружи при сохранении и загрузке;
- при изменении структуры `Game` достаточно обновить только методы `create_memento` и `set_memento`;
- соблюдается принцип инкапсуляции.

Структурно:  
- GameLoop → Game.create_memento() → Memento  
- GameLoop → SaveManager.save(memento)  
- GameLoop → SaveManager.load() → Game.set_memento()  

### Реализация без использования Memento

- `SaveManager` напрямую читает поля `game.bird_y`, `game.velocity` и т.д. при сохранении;
- `SaveManager` сам записывает значения обратно в поля `Game` при загрузке;
- при переименовании любого поля в `Game` необходимо вносить правки и в `SaveManager`;
- нарушается инкапсуляция, растёт связанность компонентов;
- сложнее изменять и поддерживать программу.

Структурно:
- GameLoop → SaveManager → game.bird_y  
- GameLoop → SaveManager → game.velocity  
- GameLoop → SaveManager → game.pipes  
- GameLoop → SaveManager → game.score

## Вывод

Использование паттерна Memento позволило:

- сохранить инкапсуляцию класса `Game`: никто снаружи не обращается к его полям напрямую;
- централизовать логику снятия и восстановления снимка внутри самого `Game`;
- упростить `SaveManager`: он хранит объекты, не зная об их содержимом;
- облегчить поддержку кода: при изменении состояния игры достаточно обновить только методы `create_memento` и `set_memento`, не трогая остальные части системы.

В результате система сохранений стала модульной, слабосвязанной и удобной для расширения.
