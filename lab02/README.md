# Лабораторная работа  
## Использование паттерна проектирования Facade

---

## Описание предметной области

В работе реализуется упрощённая система онлайн-казино с игрой в рулетку.  
Пользователь делает ставку на цвет, после чего система должна:

- проверить наличие достаточного баланса;
- сгенерировать результат вращения рулетки;
- определить выпавший цвет;
- рассчитать возможный выигрыш;
- обновить баланс игрока;
- вернуть результат игры.

Для выполнения этих действий используются несколько подсистем, что усложняет взаимодействие клиента с системой.

---

## Решение

Для упрощения архитектуры программы был применён структурный паттерн **Фасад**.  
Создан класс `CasinoFacade`, который объединяет работу внутренних подсистем:

- `BalanceManager` — управление балансом игрока;
- `Wheel` — генерация результата рулетки;
- `PayoutSystem` — расчёт выигрыша.

Клиентская часть (HTTP-сервер) взаимодействует только с методом `play()`,  
который собирает в себя всю игровую логику.  
Таким образом, сложность системы скрывается за единым интерфейсом.

## Сравнение реализации с паттерном и без паттерна

### Реализация с использованием Facade

- сервер взаимодействует только с одним классом — `CasinoFacade`;
- вся игровая логика сосредоточена в одном месте;
- внутренние подсистемы скрыты от клиента;
- уменьшается количество связей между компонентами;
- упрощается расширение функциональности.

Структурно: 
- Server → CasinoFacade → (BalanceManager, Wheel, PayoutSystem)

### Реализация без использования Facade

- сервер напрямую работает со всеми подсистемами;
- игровая логика распределена по клиентскому коду;
- увеличивается связанность компонентов;
- сложнее изменять и поддерживать программу.

Структурно:
- Server → BalanceManager
- Server → Wheel
- Server → PayoutSystem
## Вывод

Использование паттерна Facade позволило:

- сделать структуру кода более понятной;
- централизовать игровую логику;
- упростить дальнейшее возможное развитие системы.
- облегчить взаимодействие пользователя с интерфейсом, скрывая внутреннюю логику.

В результате программа стала более модульной и удобной для расширения.
